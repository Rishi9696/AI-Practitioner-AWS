## Tokens → Embeddings → Chunks (the full chain)

1. **Text → tokens.** Raw text is broken into tokens, each roughly ¾ of a word (subword tokenization handles rare/long words by splitting them into pieces, e.g. "unbelievable" → ["un", "believ", "able"]).
2. **Tokens → word/token embeddings.** Each token is looked up/mapped to its own embedding vector — an array of numbers placing that token in a high-dimensional "meaning space." This happens inside every transformer, for every prompt, every time (this is distinct from the RAG use case below).
3. **Tokens → chunk (sentence) embedding.** For RAG/semantic search, you don't keep one vector per token. An embedding model **pools** (e.g., averages) all the token vectors in a chunk into a **single vector** representing the whole chunk's meaning. One chunk = one stored vector, not a list of vectors.

**Analogy that stuck:** each token is a house with its own coordinates. A chunk is a village; the village's single location is the combined/averaged location of all its houses. The axes of that space aren't literal geography — they're abstract learned features of meaning, hundreds/thousands of them, not 2 (like lat/long).

### Word-based vs. subword tokenization — why subword wins

Word-based tokenization treats each whole word as one token ("run," "running," "runner" all get separate, unrelated entries). Two real problems follow: **vocabulary explosion** (every tense, plural, and typo needs its own slot), and the **out-of-vocabulary (OOV) problem** — a word never seen during training (new slang, a typo, a rare technical term) simply cannot be represented at all.

Subword tokenization splits rare/long words into reusable pieces — "running" → ["run", "ning"], "runner" → ["run", "ner"] — so the model recognizes the shared root "run" across all three, and can assemble an unfamiliar word out of familiar pieces instead of failing on it. This keeps the vocabulary size manageable while still being able to represent virtually any input, known or unknown. This is why every modern LLM uses subword tokenization, not word-based.

### Types of vector embeddings

- **Word embeddings (static)** — one fixed vector per word regardless of context (older models like Word2Vec, GloVe). "Bank" gets the same vector whether it means a riverbank or a financial bank — a real limitation.
- **Contextual/token embeddings** — generated by transformers, where the same word gets a *different* vector depending on the surrounding sentence, because self-attention lets neighboring words influence each token's representation. "Bank" in "river bank" vs. "savings bank" produce different vectors.
- **Sentence/document embeddings** — one pooled vector for a whole sentence, paragraph, or chunk's meaning — this is the chunk-level vector used in RAG (see word vs. sentence embedding below, and Sentence Transformers).
- **Multimodal embeddings** — map different input types (text, image, audio) into the *same* shared vector space so they're directly comparable via cosine similarity, e.g. comparing a photo against the word "dog." Amazon Titan Multimodal Embeddings and Nova Multimodal Embeddings work this way — useful for searching images using a text description.
- **Image/audio embeddings** — vectors representing visual or acoustic content, used for reverse image search or audio similarity.

For the exam, the two that matter most: general text embeddings (RAG/semantic search) and multimodal embeddings (comparing text against image/video in one space).

**Word embedding vs. sentence embedding:**
- **Word embedding** = one house's location — a single word/token's meaning in isolation (e.g., comparing "dog" directly to "puppy").
- **Sentence/chunk embedding** = the village's location — the pooled meaning of an entire passage, which is what actually gets stored and searched in a vector database for RAG. Semantic search runs on sentence/chunk embeddings, not raw word embeddings.

### Sentence transformers — the tech that actually produces those chunk vectors

A standard transformer like BERT wasn't originally designed to produce embeddings directly comparable via cosine similarity — naively averaging its token vectors together often gives mediocre similarity results. **Sentence-BERT** (the origin of "sentence transformers") modifies training so the *pooled sentence vector itself* is optimized for comparison — trained on pairs of sentences labeled similar/dissimilar, so the resulting space genuinely places similar-meaning sentences close together, measurable by cosine similarity or Euclidean distance.

Practically: sentence transformers are the family of models producing the chunk-level vectors in every RAG pipeline. When Bedrock uses Titan Embeddings or Cohere Embed to turn a chunk into its single vector, it's applying this same sentence-embedding philosophy.

**Token count ≠ embeddings.** Token count is a separate concern from meaning — it drives **cost** (billed per input/output token) and **context window usage** (capacity), unrelated to embeddings or similarity.

## Measuring similarity: Euclidean distance vs. cosine similarity

- Picture each word/chunk as an **arrow from the origin (0,0,0,...)** out to its vector's coordinates.
- **Euclidean distance** = straight-line distance between two points: `√((x1-x2)² + (y1-y2)² + ...)` — generalizes from 2D/3D to hundreds of dimensions, same formula, more terms.
- **Cosine similarity** = the **angle** between two arrows, ignoring their length. Score from -1 to 1: 1 = same direction (very similar), 0 = unrelated, -1 = opposite.
- Embedding models/vector databases (OpenSearch, Pinecone, Bedrock Knowledge Bases) default to **cosine similarity** because vector *length* can vary for reasons unrelated to meaning (e.g., how much text was in the chunk) — cosine cancels that out and only cares about direction.
- Small angle (e.g., "dog" vs. "puppy") = high similarity. Large angle (e.g., "dog" vs. "spreadsheet") = low similarity.

## Worked RAG example (2-paragraph document)

**Source doc:** Paragraph 1 about Amazon Bedrock (managed, no infra). Paragraph 2 about Amazon SageMaker (custom models, full control).

**Ingestion (done once):**
1. **Chunking** — split into Chunk A (Bedrock paragraph) and Chunk B (SageMaker paragraph).
2. **Embedding** — each chunk → one pooled vector, e.g., Chunk A → [0.9, 0.1, 0.8], Chunk B → [0.2, 0.9, 0.3].
3. **Storage** — vector database stores each vector **alongside the original chunk text** (the vector is just an index/pointer; original text is never discarded).

**Retrieval (done per user query):**
4. User asks: *"Which AWS service lets me use foundation models without managing servers?"* → embedded with the **same embedding model** → e.g. [0.85, 0.15, 0.75].
5. **Similarity search** — cosine similarity computed against every stored vector. Chunk A scores high (~0.98), Chunk B scores low (~0.40). Chunk A wins.
6. **Prompt assembly** — Chunk A's **original text** (not the vector) is inserted into the prompt as context, alongside the user's question. Chunk B is never retrieved and costs zero tokens.
7. **Generation** — the LLM answers using the retrieved chunk as grounding, reducing hallucination risk vs. relying purely on pre-trained knowledge.

**Key point:** nothing is "rebuilt." There's no reconstruction of the full document — only the most relevant chunks (by cosine similarity) are pulled and handed to the LLM as context. Query and chunks **must** be embedded with the *same* embedding model, or their vectors live in incompatible spaces and comparison is meaningless.

### Chunking strategies — how step 1 actually gets decided

Chunk size and boundaries directly affect what gets retrieved, so this is a real design decision:

- **Fixed-size chunking** — split every N tokens/characters, regardless of sentence or paragraph boundaries. Simple, predictable, but can cut a sentence awkwardly in half.
- **Fixed-size with overlap** — adjacent chunks share overlapping text (e.g., the last 50 tokens of chunk 1 also open chunk 2), so a sentence straddling a cut point still appears whole in at least one chunk.
- **Sentence-based chunking** — split along natural sentence boundaries instead of an arbitrary token count.
- **Paragraph/section-based chunking** — split along paragraph or document-structure boundaries (markdown headers, HTML tags) — this is what the Bedrock/SageMaker example above used, one coherent idea per chunk.
- **Recursive chunking** — a fallback hierarchy: try paragraph first; if still too large, split by sentence; if still too large, split by fixed size. Adapts to messy, inconsistently structured documents.
- **Semantic chunking** — use embeddings themselves to detect topic shifts (comparing consecutive sentences' similarity, cutting where similarity drops), keeping each chunk topically coherent rather than an arbitrary slice.

**Tradeoff that matters most:** chunk too large → wastes tokens, pulls in irrelevant text, dilutes the similarity signal. Chunk too small → loses surrounding context the model needs to make sense of the snippet, and multiplies the number of vectors to manage. Overlap is the standard fix for boundary-cutting problems regardless of strategy.

## Transformer architecture — self-attention

- Pre-transformer models (RNNs) read text **sequentially**, one token at a time — token 2 waits on token 1, etc. Slow to train, and long-range context tends to get lost (like a game of telephone).
- **Self-attention** lets every token weigh its relationship to **every other token simultaneously**. Example: "The trophy didn't fit in the suitcase because **it** was too big" — self-attention computes how strongly "it" relates to "trophy" vs. "suitcase" to resolve the reference.
- Because all these token-to-token comparisons happen **in parallel** (not sequentially like RNNs), they can be computed efficiently on GPUs — this parallelism is what allowed transformers to scale to billions of parameters.

### Why LLMs are called "autoregressive"

"Auto" = self, "regressive" = depending on previous values — borrowed from statistics, where an autoregressive model predicts the next value in a sequence from previous values in that same sequence.

An LLM does exactly this with text: predict token 1 from the prompt → feed token 1 back in as part of the input → predict token 2 from prompt + token 1 → feed that back in → predict token 3 from prompt + token 1 + token 2 → and so on. Each new token depends on (regresses on) everything generated so far, **including the model's own prior outputs**. This is the direct cause of why generation is sequential and why output length drives latency (covered below) — token 47 cannot exist until tokens 1–46 do, since the model needs them as input to produce it.

### What pre-training actually teaches the model

Pre-training is self-supervised: no human labels anything, the model is just shown massive raw text and given one repeated task — predict the next token given everything before it. Getting good at that one task, across trillions of tokens, forces the model to implicitly pick up grammar and syntax, factual associations ("the capital of France is" → "Paris"), reasoning and argument structure, style/tone, and even rudimentary logic — none explicitly taught, all emerging as a side effect of next-token prediction.

**Exam nuance:** pre-training does not teach task-specific behavior like "be a helpful, polite chatbot" — that comes later from fine-tuning (including RLHF). Pre-training gives broad general language/world knowledge; fine-tuning shapes behavior on top of it.

### Post-training and fine-tuning — turning raw knowledge into a usable assistant

**Post-training** is the umbrella term for everything done *after* pre-training to shape behavior rather than knowledge. Pre-training answers "does it understand language and facts?"; post-training answers "does it behave the way we want?" It's far smaller and cheaper than pre-training (thousands–millions of examples, not trillions of tokens), since it builds on what's already learned rather than starting from scratch.

- **Supervised fine-tuning (SFT) / instruction fine-tuning** — trains on curated (instruction, ideal response) pairs so the model learns to actually *follow* an instruction in a given tone/format, rather than just continuing the sentence. Example: a base model might complete "Write a haiku about the ocean" by rambling on about oceans; SFT teaches it that an instruction expects a direct, on-task response.
- **Domain-adaptation fine-tuning** — different goal: teaches specialized knowledge/vocabulary, not response behavior. Continues training on unlabeled domain text (e.g. clinical notes, legal filings) so the model absorbs domain terminology and patterns — no labeled instruction/response pairs needed, just more raw text from the target domain.
- **RLHF (Reinforcement Learning from Human Feedback)** — layered on after SFT to align behavior with human preference, not just correctness. The model generates multiple candidate responses; humans rank them; a reward model is trained to predict those rankings; the LLM is then fine-tuned via reinforcement learning (commonly PPO) to produce responses that score higher against that reward model. This is the step that makes a model feel genuinely helpful/careful/safe, not just technically instruction-following.
- **Parameter-efficient fine-tuning (PEFT), e.g. LoRA** — full fine-tuning updates every weight (expensive, pre-training-scale compute). PEFT freezes the base model and trains only a small set of added "adapter" weights on top — cheaper, faster, and swappable across tasks on the same frozen base. This is generally what "fine-tuning" means in practice on managed platforms like Bedrock's custom models.

**Exam customization order (cheapest → most expensive):** prompt engineering → RAG (no training, just retrieval) → instruction fine-tuning (behavior/format/tone) → domain-adaptation fine-tuning (deep specialized knowledge) → training from scratch (essentially never justified). RLHF sits alongside/after instruction fine-tuning for alignment rather than raw task-following. **Any fine-tuned/custom model on Bedrock requires Provisioned Throughput to host** — on-demand doesn't support custom models.

## Diffusion models — text-to-image

- **Forward diffusion (training):** the model watches real images get progressively buried in **random noise** (not blank/white — actual random static, e.g. Gaussian noise) across many steps, and learns to predict what noise was added at each step.
- **Reverse diffusion (generation):** start from **pure random noise**, then repeatedly remove noise step by step — guided by the text prompt — until a coherent image emerges matching the description.
- **Why not start from a fixed image (e.g., all-white)?** If the starting point were fixed and the process deterministic, the same prompt would always generate the identical image — no variety. Random noise as the starting point is what gives different outputs for the same prompt.
- **Seed parameter:** controls the exact random noise pattern used as the starting point. Same prompt + same seed = reproducible image. Same prompt + different/no seed = varied outputs.
- Exam default answer for "how does an image-generation model work" → **diffusion** (Stable Diffusion, Amazon Nova Canvas).

## Sampling controls vs. latency/cost (heavily tested distinction)

At each generation step, the model computes a probability distribution over its whole vocabulary for "what token comes next."

- **Temperature** — reshapes how peaked vs. flat that distribution is. Low temperature → sharpens toward the highest-probability token (focused, repeatable — good for factual/code tasks). High temperature → flattens the distribution, giving lower-probability tokens a real chance (creative, varied — good for brainstorming/poetry).
- **Top K** — restricts sampling to only the K most likely next tokens (fixed count), discarding everything else before sampling.
- **Top P (nucleus sampling)** — keeps adding next-most-likely tokens until their combined probability crosses threshold P, then samples only from that pool. Adapts pool size to model confidence automatically.
- **None of these three affect price or latency.** They only change *which* token gets picked, not how much computation happens.
- **Latency** is driven by: **model size** (bigger = more computation per token) and **token count**, especially **output length**, because generation is strictly **sequential** — token 47 requires tokens 1–46 to already exist. Input length adds some latency too, but output length dominates since input can be processed all at once while output must be generated step by step.
- **Exam trap:** "reduce latency" → smaller model or shorter output, **not** lower temperature. "Does raising Top P increase cost?" → **no**, cost is 100% about token counts.

## Bedrock pricing modes — with scenarios

| Mode | Mechanism | When it fits | Key trap |
|---|---|---|---|
| **On-demand** | Pay-as-you-go: text models per input/output token, embedding models per input token, image models per image generated. No commitment. | Unpredictable/low volume, prototyping (e.g., an internal test chatbot with sporadic traffic) | — |
| **Batch** | Submit many prompts at once as a background job; results written to a single S3 file. Not real-time. | Large offline workloads with no immediate-response requirement (e.g., summarizing 50,000 support tickets overnight) | **Up to 50% discount** — this is the "cheap" answer |
| **Provisioned Throughput** | Commit to a term (1 or 6 months), buy "model units" guaranteeing max tokens/minute, reserved exclusively for you. | Steady high volume needing guaranteed performance; **mandatory** for hosting fine-tuned/custom models | **Not a cost saver** — it's a capacity/performance guarantee. Often *more* expensive for low/bursty volume. |

**One-line exam framing:** "discount" → batch. "Guaranteed capacity" or "custom model" → provisioned throughput. "Unpredictable, pay only for use" → on-demand.

## Context engineering vs. prompt engineering

- **Prompt engineering** = crafting the wording of a single instruction to the model.
- **Context engineering** = the broader decision of **everything the model sees at inference time** — system instructions, retrieved documents (RAG), conversation history, tool definitions, desired output format. Prompt engineering is one ingredient inside it.
- **Goal: relevance over volume.** The context window is finite and every input token is billed, so the aim is the *right* context, not the *most* context.
- **Bad example:** dumping an entire 50-page policy PDF + 6 months of chat history + every tool definition into a prompt for a simple refund question → wastes tokens, and can actually cause **hallucinations** (irrelevant text drowns the useful signal, model blends unrelated details into a plausible-but-wrong answer).
- **Good example:** retrieve only the 2–3 relevant policy chunks (via RAG), include only recent relevant conversation turns, a short targeted system instruction, and only the tool definitions relevant to this query.
- **Failure mode to recognize:** a long conversation that "loses track" or contradicts itself is usually a **context engineering problem** (relevant earlier content got pushed out or was poorly organized), not a model-intelligence problem. Fixes: summarization/compaction of older turns, long-term memory for persisted facts across sessions, or retrieval over chat history instead of replaying the whole transcript — a bigger context window alone is a band-aid, not a fix.

## Agentic AI — concepts tied together with one running example

**Running example:** an IT support agent handling "I'm locked out and also need to request a new laptop."

| Concept | What it does in the example |
|---|---|
| **Core loop (perceive → plan → act → observe → iterate)** | Recognizes two subtasks (unlock account, file laptop request), plans an order (verify identity → reset password → file ticket → confirm), acts, checks each step actually succeeded before moving on — not a single one-shot reply like a plain chatbot |
| **Chain of thought** | Internally reasons "verify identity first since this is security-sensitive, then reset, then check ticket requirements" before calling anything |
| **Tool use / action groups** | Calls real functions/APIs: `reset_password(user_id)`, `create_ticket(type='hardware')` — reasoning alone can't reset anything; Bedrock Agents call these action groups |
| **RAG (as a tool)** | If it needs to check laptop-request policy for the user's role, it retrieves the relevant policy chunk from a knowledge base rather than guessing — same embeddings/vector-DB mechanism as the document RAG example above |
| **Memory** | Short-term: remembers the employee ID given 3 messages ago, within this session. Long-term: recalls next month, in a new session, that this user's last request was a MacBook, engineering role |
| **Orchestration** | Manages step order, handles errors (e.g., retry a failed API call once, then escalate to a human), passes verified identity from step 1 into the ticket filed in step 3 |
| **MCP (Model Context Protocol)** | Standardized connector to the ticketing/password systems — build once, any MCP-compatible agent can reuse it, instead of custom one-off integration code per agent |
| **Multi-agent systems** | For a bigger goal like "onboard new employee," a supervisor/orchestrator agent decomposes into subtasks and routes to specialist workers (account provisioning, hardware request, building access), then merges results — mirrors a manager delegating to specialists |

**One-line summary:** an agent is a *loop*, not a single response — plan, act via tools, observe, repeat — supported by memory for context, orchestration for reliability, MCP for standardized tool connections, and multi-agent patterns when the task is big enough to split across specialists.
